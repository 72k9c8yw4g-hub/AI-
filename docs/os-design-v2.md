旧設計書4冊(憲法 v3.1 / 運用 v2.0 / 技術 v1.3 / 実装準備 v1.0)と、実際に動いている実装

- **この1冊が唯一の設計正典**。旧4冊は歴史資料(思想の原本)として尊重するが、矛盾したらこちらが勝つ。

## 第1章 監査結果: 旧設計の良かった点(そのまま継承する)

実際に数週間ぶん実装・運用して、**設計として正しかった**と確認できたもの。v2でも不変。

1. **憲法の思想そのもの** — 対等な共同創業者 / YESマン禁止 / 最終決定権はユーザー /
   事実と意見の分離 / 実行優先(考えるだけ禁止)。この製品の魂であり、類似アプリに無い独自価値。
2. **無承認保存禁止** — AIは「保存候補」までしか作れず、決定になるのはユーザーが[承認]した時だけ。
   AIの暴走を構造で防ぐ、最重要の発明。
3. **決定の進化追跡(supersedes チェーン)** — 決定は消さない。新決定が旧決定をアーカイブし、
   変更理由を残す。「なぜ変わったか」が後から辿れる。Dscribe側の記憶とも同じ思想で一貫。
4. **独立監査・非中継** — 監視官はデータフローに割り込まず、横から警告だけ出す。
   実装上も monitor 発言はメンターの文脈から除外されており、アーキテクチャとして綺麗。
5. **作業AIの固定3ターン収束** — 無限ループをルールではなく構造で防ぐ。実用的。
6. **成果物はメンター経由** — ユーザーへの声を一本化する。混乱しない。
7. **役割ごとにモデル/APIキーを分けられる** — 設計時点では過剰に見えたが、
   無料枠の分散(Gemini制限対策でGroq/Cerebrasに逃がす)で**実際に効いた**。先見性があった。
8. **優先順位の思想**(憲法>運用>技術>実装) — 思想がUI都合で歪められるのを防いだ。
   ※ただし「4冊に分かれているから優先順位が必要になった」という側面もある(第2章-9)。

## 第2章 監査結果: 旧設計の悪かった点(修正 or 廃止)

実運用で問題が**実際に起きた**ものと、設計内部の矛盾。各項に判定を付す。

| # | 問題 | 根拠 | 判定 |
|---|---|---|---|
| 1 | **監視官が毎送信で走る設計はコストと誤警告の温床** | 1送信=LLM2回(メンター+監視官)。的外れ警告が実際に発生しユーザーが指摘(「的外れなこといわないでほしい」)。プロジェクト絞り込み等で抑えたが、根本は頻度設計が無いこと | **修正**(→第4章: イベント駆動化を提案) |
| 2 | **監査タイプ9種は過剰** | 憲法v3.1で法律/規約/品質/セキュリティまで拡張したが、LLMの確信度が低い分野ほど誤警告になる。drift は設計自身が「基本使わない」と書いている=最初から不要だった | **修正**(コア3種を既定、拡張はオプトイン。drift廃止) |
| 3 | **「決定」と「記憶」の二重概念でユーザーが混乱** | OSの決定=Dscribeのmemoriesという共有は良い設計だが、UIに「決定事項」「保存データ」タブが並び、ユーザー自身が「Dscribeとまざってね?」と混乱した(実話) | **修正**(→第6章: 「記録」に用語一元化+タブ統合を提案) |
| 4 | **ナビ8ボタンは多い** | ホーム/チャット/決定/保存/ファイル/検索/AI会話/設定。スマホ下タブとして限界を超えている。使用頻度に大差がある | **修正**(→第6章: 5タブ案) |
| 5 | **メンター確認ステップ(運用8章)は官僚的** | 📝提案のたびにLLM2回消費(記録官+メンター所見)。多段承認は「実行優先」の憲法と緊張関係にある | **修正**(既定は記録官のみ、メンター所見はオプトイン化を提案) |
| 6 | **PC4タブ同時表示(実装11章)** | 「スマホ主対象」という同じ設計書の前提と矛盾。価値に対して大改造 | **廃止** |
| 7 | **共同編集 owner/editor/viewer(実装13章)** | 「1アカウント=独立した1つの脳」という基本思想と根本的に矛盾。やるなら別プロダクト | **廃止** |
| 8 | **プロジェクト複製(実装16章)/完全復元(実装19章)** | 1人利用で使途が薄く、supersedes整合の複雑さに見合わない | **廃止** |
| 9 | **4冊分割そのものが矛盾の温床** | 冊間の矛盾(上記6,7)が現に生まれ、解決に「優先順位ルール」という追加ルールが必要になった。設計書は読者が1人(＝自分とAI)なのだから1冊でいい | **修正**(本書に統合。旧4冊は凍結) |
| 10 | **モデル進化への追従が設計に無い** | 設計書はモデル名を固定で書く前提だったが、実際はモデル名の廃止・レート制限が頻発。実装側で「モデル名自己修復」「自動フォールバック」を発明して吸収した | **修正**(第7章で正典化) |
| 11 | **セキュリティ章が薄い** | トークンinURLの脅威モデル、キーの暗号化、CSP、コスト上限(暴走保険)は全部実装側の発明。設計書に無かった | **修正**(第7章で正典化) |
| 12 | **節目レポートの起動条件が未設計** | 手動ボタンは押す動機が弱く、毎回自動はうるさい。「いつ出すか」が設計されていなかった(現状: 手動+承認時自動のオプトイン) | 保留(現状維持。使用実績を見て判断) |
| 13 | **「アプリ化」の定義が曖昧** | 実装準備書の「アプリ」はネイティブを匂わせるが、実体はPWA。ユーザーも一度「Chromeの無理やりアプリでは?」と看破した。誠実に「PWAである」と明記すべき | **修正**(本書で明記: 本製品はPWA。ネイティブ化はやらないことリスト入り) |

## 第3章 憲法(v2・7原則に凝縮)

旧憲法10則+運用ルールから、実装で本当に効いている原則だけを残した。**この章は今後も変えない**。

1. **対等な共同創業者** — AIはユーザーの部下でも上司でもない。YESマン禁止。
   根拠ある限り反対してよいが、根拠が崩れたら直ちに撤回する。反対の目的化も禁止。
2. **最終決定権はユーザー** — AIは決めない。ユーザーが最終判断を宣言したら議論を止め、実行に移る。
3. **無承認保存禁止** — AIが作れるのは「候補」まで。決定はユーザーの明示的な[承認]でのみ確定する。
4. **決定は消さない** — 変更は supersedes による進化として記録し、理由を残す。削除ではなくアーカイブ。
5. **事実と意見の分離** — 確認済みの事実と推測・意見を必ず区別して話す。
6. **実行優先** — 議論→結論→実行→検証→改善。無限議論・無限調査・先送りをしない。
7. **監査は独立・非中継** — 監視官は会話に割り込まず、決定も指示もしない。警告のみ。

## 第4章 AI組織 v2(4役職+ユーザー)

### 現行のコスト構造(事実)
| 操作 | LLM呼び出し | 内訳 |
|---|---|---|
| 送信 | 2 | メンター + 監視官 |
| 📝決定を記録 | 2 | 記録官 + メンター所見 |
| 🛠委任 | 4 | 作業AI×3ターン + メンター整理 |
| 📋レポート | 1 | 監視官 |
| 上限 | 500/日 | 暴走・トークン漏洩時の保険 |

### 役職定義(実体は `src/os/org.ts` の各 SYSTEM プロンプト。迷ったらコードが正)
- **メンター兼司令塔** — 主対話相手。短文中心・1メッセージ1論点(長文は仕様書・最終報告等のみ)。
  プロジェクト開始時は目的/現状/ゴールを確認してから走る。
- **記録官** — 会話から結論を最大1件抽出して候補化。summary必須。supersedes判定を行う。保存はしない。
- **特命監視官** — 警告のみ。**既定は警告しない(空配列)**が大原則。確信の無い警告は出さない。
  監査対象の決定はそのチャットのプロジェクトに限定(誤警告対策・実装済み)。
- **作業AI群(1/2/3)** — 草案→批判レビュー→最終化の固定3ターン。相談必須事項(不可逆操作・
  コスト発生・目的逸脱の疑い)は勝手に進めず必ず戻す。成果物はメンター経由。

### 【提案】監視官のイベント駆動化(未実装・承認待ち)
毎送信で走らせるのをやめ、次のときだけ走らせる:
- ユーザー発言が5回たまったとき(まとめて監査)
- 📝候補が作られたとき(決定前の最終チェックとして最も価値が高い)
- ユーザーが明示的に呼んだとき(📋)

効果(推定): 通常会話のコストが半減(2→1)。文脈がまとまってから監査するため的外れ警告も減る。
リスク: 脱線への即時警告は失われる。→ ユーザーの実感として「即時性」と「静かさ」どちらが大事かで決める。

### 【提案】監査タイプの整理(未実装・承認待ち)
- 既定ON: `deviation`(脱線) / `loop`(堂々巡り) / `contradiction`(既存決定との真っ向矛盾)
- オプトイン(⚙️設定): `legal` / `tos` / `quality` / `security` / `inefficiency`
- 廃止: `drift`(設計自身が「基本使わない」と認めていた)

## 第5章 データモデル(ほぼ現状維持)

- 基盤: Cloudflare Workers + D1。Dscribe と**同一Worker・同一DB・同一アカウント**(Route C)。
- **OSの決定 = Dscribe の memories(kind=decision)**。これは意図的な共有であり、バグではない。
  ユーザーがどちらの画面から見ても同じ「脳」を見ている——これが製品の核心。
- supersedes チェーン(`supersedes_id`/`superseded_by_id`/`supersede_reason`)で決定の進化を線形に記録。
- 候補は `os_candidates`(pending/approved/rejected + reject_reason + mentor_note)。
- ファイルは D1 に base64(≦約525KB。画像はブラウザ側で1600pxに縮小)。R2不要のカード無し構成。
- 週次バックアップ(R2任意)+手動エクスポート+追記復元。AI会話ログは50日でパージ。

### 【提案】用語の一元化(未実装・承認待ち)
「決定」「記憶」「メモ」「ノート」が画面ごとに揺れている。ユーザー向け語彙を
**「記録」(そのうち種別が 決定/メモ/記憶)** に統一する。データ構造は変えない。表示だけ。

## 第7章 技術基盤(実装側の発明を正典化)

旧設計書に無かったが、実運用で必須と判明したもの。v2から正式な設計要件とする。

1. **スタブ応答モード** — APIキー未設定でもアプリは動き、スタブと明示した応答を返す。
   初期設定の壁を下げ、UI検証をキー無しで可能にする。
2. **モデル名の自己修復と自動フォールバック** — モデル名の廃止・レート制限(429)は正常事象として
   設計に織り込む。既定モデルはコード側で更新し、保存済み設定が壊れていれば自動で現行に置換。
3. **コスト上限** — 1日500呼び出しで遮断。トークン漏洩時の焼き尽くし対策を兼ねる。
4. **APIキーの暗号化保存** — D1に平文で置かない(AES-GCM、KEKはSecret)。チャットにキーを書かない運用。
5. **キー解決の優先順位** — 役割キー → 共有プロバイダキー → 環境Secret。
6. **本製品はPWAである** — インストール可・オフラインシェル可。ネイティブアプリではないと明記する。
7. **冪等マイグレーション** — スキーマ変更は `CREATE TABLE IF NOT EXISTS` + try/catch `ALTER` で、
   稼働中の本番DBに初回リクエスト時に自動適用される方式を守る。
