# AI意思決定OS 実装ロードマップ（ルートC: 独立アプリ本建築）

4つの設計書（憲法 v3.0 / 運用 v2.0 / 技術 v1.1 / 実装準備 v1.0）を、動く単位で積む。
既存の Dscribe（Cloudflare Workers + D1）と**同じ Worker 内**に建て、Dscribe の
記憶・決定ロジックは「記録官モジュール」として再利用する。デプロイ先は1つのまま。

- OS アプリ画面: `GET /os/<トークン>`（ChatGPT型・スマホ主対象）
- OS の REST: `/api/<トークン>/os/...`
- 既存 Dscribe ダッシュボード `/app/<トークン>` は当面「記録ビュー」として併存

## 役職 → 実装対応

| 役職（憲法） | 実装 |
|---|---|
| ユーザー | 人間。最終決定権・承認 |
| メンター兼司令塔 | LLM ロール。共同創業者・YESマン禁止・反対権・実行優先 |
| 特命監視官 | LLM/ルール。ループ・逸脱・矛盾を横から監査し警告（中継しない） |
| 記録官 | 保存候補生成 → 承認 → 決定事項(Active/Archived)。Dscribe を転用 |
| 作業AI群 | メンターがアサイン。役割別モデル可。AI会話ログは閲覧専用 |

## フェーズ

### Phase 1 — 背骨（チャット + メンター）★実装中
- スキーマ: `os_chats` / `os_messages` / `os_role_config`
- プロバイダ抽象: 役割 → モデル(anthropic/openai/gemini)。キーは Secret。未設定はスタブ応答
- メンターのシステムプロンプト（憲法・運用から生成）
- `/os/<token>` に ChatGPT型チャットUI（発言者ラベル付き）
- ユーザー発言 → メンター応答の1往復が回る

### Phase 2 — 記録官 + 承認フロー
- 保存候補カード（種別・タイトル・内容・タグ・[承認][却下]）
- 承認で決定事項 Active 化 / 旧決定を Archived（supersedes 転用）
- 決定事項画面: Active / Archived / 承認待ち、検索・タグ・関連決定・元チャット追跡
- 無承認保存の禁止・明示的承認ルール（「保存/承認」はOK、「OK/いいね」は不可）

### Phase 3 — 監視官 + 作業AI
- 監視官: 話題逸脱・無限ループ・矛盾・離脱傾向を検知して警告（節目でレポート）
- 作業AI群: メンターがアサイン、AI同士会話ログ、成果物はメンター経由で提示
- 役割別モデル（メンター=Gemini / 監視官=Claude / 記録官=GPT など）の設定画面

### Phase 4 — 共同編集・バックアップ・仕上げ
- 権限: オーナー / 共同編集者 / 閲覧者
- バックアップ/復元（ZIP、システム全体 / プロジェクト単位、自動+手動）
- ファイル管理・プロジェクト複製・通知

## 設計上の確定事項
- 憲法 > 運用 > 技術 > 実装準備 の優先順位を守る（矛盾する実装を入れない）
- 決定事項は削除しない。旧版はアーカイブ参照専用（憲法第8章・supersedes）
- 監視官はデータフローを中継しない。独立監査ラインから警告のみ
- 保存はユーザーの明示的承認が必須（運用第8章）
